# AI Platform Development Rules

## 1. 目的

本ドキュメントは、AI Platformの開発における共通ルールを定義する。

本プロジェクトでは、AIによる設計支援、自動実装、テスト、デバッグ、記録を前提とする。そのため、人間とAIが同じ基準に従い、安全性、再現性、保守性、レビュー可能性を維持しながら開発を進めることを目的とする。

## 2. 基本原則

以下の原則を遵守する。

- 人間が最終的な意思決定者となる
- 小さな単位で変更する
- 常に動作確認可能な状態を維持する
- 設計変更は実装前に文書へ反映する
- 変更内容、判断理由、実行結果を記録する
- 不明点を推測だけで補完せず、必要に応じて確認する
- レビュー可能な単位で実装する
- 特定のAIサービスへ過度に依存しない

## 3. 役割分担

### 3.1 ChatGPT

ChatGPTは、主に設計、整理、レビュー、説明を担当する。

#### 担当範囲

- 要件整理
- 完了条件の定義
- システム設計
- アーキテクチャ検討
- 技術選定支援
- ドキュメント作成
- Claude Code向け実装指示の作成
- 実装結果のレビュー
- 学習支援と技術説明

#### 原則として行わないこと

- ローカルファイルの無承認変更
- 危険なコマンドの自動実行
- ユーザー承認を伴わないデプロイ判断
- 不明な情報を事実として扱うこと

### 3.2 Claude Code

Claude Codeは、主に実装、テスト、デバッグ、ローカル操作を担当する。

#### 担当範囲

- リポジトリ調査
- ソースコード実装
- ファイルの読み書き
- リファクタリング
- テスト実行
- Lint、Format、Build
- エラー調査とデバッグ
- Git差分の作成
- 作業結果の報告

#### 実装前に必ず提示する内容

- 調査結果
- 変更対象
- 実装方針
- 影響範囲
- リスク
- テスト方針

### 3.3 Orchestrator

Orchestratorは、AI Platform全体の実行制御を担当する。

#### 担当範囲

- タスク分解
- 担当AIやツールの選択
- 実行順序の管理
- 状態管理
- 依存関係管理
- 承認待ちの制御
- 再試行
- エラー処理
- 作業ログ保存

## 4. 標準開発フロー

作業は原則として以下の順序で進める。

1. ユーザーが目的を伝える
2. ChatGPTが要件と完了条件を整理する
3. 必要に応じて設計書を更新する
4. Claude Codeが対象リポジトリを調査する
5. Claude Codeが実装計画とリスクを提示する
6. ユーザーが計画を承認する
7. Claude Codeが実装する
8. テスト、Lint、Buildを実行する
9. ChatGPTまたは人間が結果をレビューする
10. GitHubへ変更、ログ、決定事項を保存する
11. 次のアクションを整理する

承認が必要な変更については、ユーザー承認前に実装を開始しない。

## 5. 変更管理

- 一つの変更では、一つの目的に集中する
- 大規模変更は複数のタスクへ分割する
- 既存機能を変更する場合は、影響範囲を明記する
- 不要なリファクタリングを同時に行わない
- 既存仕様と異なる実装を行う場合は、先に設計書を更新する
- 生成物や一時ファイルを不要にコミットしない

## 6. Git運用

### 6.1 ブランチ

- `main` ブランチへ直接コミットしない
- 作業ごとにFeature Branchを作成する
- ブランチ名は内容が分かる形式にする

例:

```text
feature/add-task-orchestrator
fix/logging-error
 docs/update-architecture
```

### 6.2 コミット

- コミットメッセージは英語で記述する
- 一つのコミットでは一つの論理的変更を扱う
- 意味の分からないメッセージを使用しない

例:

```text
feat(orchestrator): add task state management
fix(logging): handle missing log directory
docs: update development rules
```

### 6.3 Pull Request

- 原則としてPull Requestを作成する
- 変更目的、変更内容、テスト結果、リスクを記載する
- 関連Issueがある場合は紐付ける
- レビュー前に自己確認を行う

## 7. テストと品質

- 実装後は必ず関連テストを実行する
- テスト不能な変更は、理由と代替確認方法を記録する
- 既存テストを無断で削除しない
- テスト失敗を無視して完了扱いにしない
- Lint、Format、Buildがある場合は実行する
- バグ修正では、可能な限り再発防止テストを追加する

## 8. セキュリティ

以下を禁止する。

- APIキー、パスワード、秘密鍵、トークンをGitへ保存すること
- `.env` や認証情報をコミットすること
- 承認なしでファイルやリソースを削除すること
- 承認なしで外部サービスへデータを送信すること
- 承認なしで本番環境へデプロイすること
- ホームディレクトリ全体やディスク全体を常時許可すること
- 機密情報をログへ記録すること

Claude Codeには、原則として対象プロジェクト単位でアクセスを許可する。

以下の操作は明示的な承認を必須とする。

- ファイル削除
- 大量上書き
- 外部送信
- パッケージの大規模更新
- データベース変更
- クラウドリソース変更
- デプロイ
- Gitの強制Push

## 9. ドキュメント管理

設計変更を伴う場合は、以下の順序で更新する。

1. Architecture
2. Development Rulesまたは関連仕様書
3. Implementation
4. Test
5. Operation Log

実装だけを先行させない。

主要ドキュメントは `docs/` 配下で管理する。

## 10. ログ管理

各作業で以下を記録する。

- 実施日時
- 目的
- 要件
- 完了条件
- 実施内容
- 判断理由
- 実行したコマンド
- 変更ファイル
- テスト結果
- 発生した問題
- 解決方法
- 未解決事項
- 次のアクション

## 11. エラー対応

エラーが発生した場合は、以下の順序で対応する。

1. エラー内容を記録する
2. 再現条件を確認する
3. 原因候補を整理する
4. 影響範囲を確認する
5. 修正案を提示する
6. 承認後に修正する
7. 再テストする
8. 解決方法を記録する

同じ操作を根拠なく繰り返さない。

## 12. 完了条件

タスクは、以下を満たした場合に完了とする。

- 要件を満たしている
- テストが成功している
- 変更内容がレビュー可能である
- 必要なドキュメントが更新されている
- セキュリティ上の問題がない
- 未解決事項が明示されている
- 次のアクションが整理されている

## 13. AI Platformの開発思想

AIは人間の判断を置き換えるものではなく、思考整理、実装加速、品質向上、記録自動化を支援するパートナーとして利用する。

AIによる自動化の範囲を広げる場合でも、重要な判断、危険な操作、外部公開、本番反映については、人間が最終承認を行う。

## 14. 言語・命名規則

GitHub全体の共通言語・命名ポリシーは、以下を正本とする。

~~~text
YDTNK/00-ai-start-here/docs/language-policy.md
~~~

AI Platformでは、以下をプロジェクト固有の適用ルールとする。

- README、Architecture、Roadmap、Runbook、運用文書、Issue本文、Pull Request本文は日本語で記述する
- Branch名、Commit message、Issue・Pull Requestタイトル、ファイル名、コード識別子、設定キー、Intent、Statusなどの固定値は英語で記述する
- 構造化データではキーと固定値を英語、人間向けの説明と判断理由を日本語にする
- 製品名、CLIコマンド、API名、規格名は原文を維持する
- 既存英語文書は実質的な更新機会に整え、言語統一だけを目的とする大規模変更や情報削除を行わない
- 共通ポリシー全文はこのRepositoryへ複製しない
- 例外が必要な場合は、理由をIssueまたはPull Requestに記録する
